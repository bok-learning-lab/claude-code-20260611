# Day 2 (9 June 2026) — Top 10 Key Takeaways

_Distilled from the diarized session transcript ([`20260609-transcript.md`](../../day-2-recap/20260609-transcript.md)), Chunks 1–9 (~14:03–15:26). Instructors: Marlon Kuzmick (MK) and Madeleine Woods (MW), with Marlon leading the repo tour and Becca — the Learning Lab's executive director — giving the conceptual Git explainer. Faculty projects from Jordan, Jungyoon, Lindsay, and others._

---

1. **Day 2 is the bridge: from chat and Cowork into Claude Code — and the desktop app's Code panel is the democratic on-ramp.** "Today we are finally crossing the bridge from chat and Cowork into Code" — flick to the third panel on the right. You can do "ninety-nine percent of everything Claude can do" right in the desktop app's Code tab, so that's where the whole room stays; developers who want the CLI / VS Code (IDE) path can set it up after class (~30 min). New names for the same idea: a new *chat* (Chat), a new *task* (Cowork), a new *session* (Code) — "a new jotter essentially."

2. **Three ways to get the repo onto your machine — and the easiest is to just ask Claude to clone it.** Download the green **Code → zip** and unzip it; run `git clone <url>` in the terminal; or, in the desktop app, create a folder, select it, and say *"can you clone this repo here?"*, then paste the URL and hit allow. One catch: copy the URL from the green **Code** button's clipboard icon, not the plain link in the email — "it's not smart enough to change the syntax" otherwise. Mac and PC all do the same thing now.

3. **One dedicated `Development` folder — never your desktop or root drive.** The Day-1 discipline carries straight into Code: make one folder you'll remember (most people made a `Development` folder), clone into it, and point Claude there. "I wouldn't give it access to my desktop or, you know, my root drive." Cowork and Code ask permission per file, so a contained folder keeps that manageable.

4. **Becca's conceptual Git: Google Docs version history, but for a whole project.** Git was built by coders to track versions of code they develop collaboratively — but it turns out to be "a very general system for keeping track of collaborative work that has multiple versions." The analogy: how useful Google Docs version history is when you work with someone — except for an entire folder/directory of files, tracking every change *and* the human comments about *why* a batch of changes was made.

5. **Three reasons Git matters even if you never write code.** (1) **History** — go back and pull something out of your own project's past. (2) **Claude reads it first** — "Claude is built to look very first thing at the Git repository in order to understand the history of the project," a way around the limited context window across sessions. (3) **Safety net** — when generative AI changes a file in a direction you didn't intend, a tracked repo makes it "extremely easy to go back to the earlier versions and pull those back out" so you don't lose your work.

6. **How Git actually works: a hidden `.git` folder, snapshots, and a remote on GitHub.** Editing files does nothing until you decide *"now's the moment to take a picture"* — you choose which changes to track, add a message saying what they mean, and **commit**. GitHub is "a remote repository out on the internet which stores a copy of what's locally on your machine" — private or public — a second layer of protection and the way you share. Even if you deleted your local `.git`, the GitHub copy survives.

7. **`.gitignore` keeps secrets out — never push API keys or passwords.** Some files (an API key, a password, proprietary or very large files) are "really only for you." Don't put them in your chats with Claude, and definitely don't push them to GitHub. A special **`.gitignore`** file tells Git never to include them — and you can just ask Claude to make one.

8. **You don't learn Git commands — Claude is fluent, and "Create PR" basically means commit.** Claude Code surfaces buttons (create a repo, commit, push), or you just ask in plain language and "Claude will know exactly what you're talking about." The one bit of jargon: it offers to **"create a pull request,"** which "essentially means create it, commit it, and pull it onto the main branch." The on-screen counter of additions/subtractions since your last commit is a nudge — **commit often**; each commit is a more granular memory and a point you can jump back to.

9. **Even solo, you're now doing collaborative work — because agents commit on your behalf.** Ask Claude to "summarise the work people did on this branch today" and it shows every commit down to the minute, who made it, and which branches merged in ("Animal catacomb is Madeleine's alter ego"). Picture it as a **tree**: a contributor's work arrives as a **branch**, a pull request "knocks on the door of the main repository," and you open the door, review, and merge — so many people (or many Claude agents, or a roomful of TFs generating text) can work without overwriting each other. Live demo: a first-time contributor's spelling-fix PR merged in, then `git pull` (or "hey Claude, there was a change on GitHub, can you pull?").

10. **Point Claude at one file by its path — and remember it's all just text, "turtles all the way down."** The core move: click the **three dots** next to a file → **copy path** → paste it into your prompt ("the path is the unique path to that resource on your computer," like a URL on the web). Teaching point: students could query a transcript you've vetted — *"top five takeaways from Day 1"* — and ask their own questions of good-quality content. In the Code tab you can also **edit files yourself**, with your own fingers (unlike chat/Cowork artifacts, which you must keep asking Claude to change) — what you want for "intellectual work you care about." And be efficient: handing over a specific path is far cheaper than making Claude search the whole repo against your 200K–1M-token window.

## Secondary points worth keeping

- **The Homebrew tax for some Macs.** Git was missing on some older Macs; the fix is to install **Homebrew** (`brew.sh`, "a package manager for Mac"), run the two lines it prints out, then `brew install git`. In Terminal your password types invisibly — "use the force and just keep typing." A one-time annoyance, then you're in good shape.
- **Find your files in the Code tab.** The views icon (top-right) only appears *after* you ask a question; click it → **Files** to see the whole repo. Code editors like VS Code show files by default; in Claude Code you have to nudge it.
- **`.docx` / `.xlsx` are proprietary, not text.** Word and Excel files are Microsoft's own formats; Claude can read them in and write them out via skills, but everything works best converted to **simple text / markdown** — "strings in, strings out" wants the simplest text form. A folder of PDFs or `.docx` files can be batch-converted.
- **Recipes are remixable starting points.** Fork one: copy its path and say *"make me a folder using this scaffolding in faculty-recipes for [my course], and search the internet for some starter papers."* Shown live: the **gradebook visualization** (a grade-explorer HTML on synthetic data), a **video word-frequency search** (the Shia LaBeouf "just do it / don't let your dreams be dreams" corpus, hard-coded in ~9 minutes), a reworked **Math** course, and a pared-down **bioengineering** course pack.
- **Keep big stuff local, share small stuff.** A full Canvas download (PowerPoints, videos) can be worked on locally but needn't be pushed to GitHub — the shared course packs were deliberately kept small so they were easy to send.
- **HTML opens in the browser.** Any `.html` file in the repo can be viewed in Claude or by pasting its path into Chrome/Edge; later days will cover publishing it so students can see it.

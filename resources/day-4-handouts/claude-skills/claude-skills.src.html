Day 4 · Skills

Skills: A Field Guide where they live, how to arrange one in a repo, and how to write the SKILL.md

A skill is a folder containing a file named SKILL.md — a conditional prompt with a header. Every session, Claude loads just the headers of every skill it can see (a map of your prompt library); when your request matches a description, the full body is read in and followed. Header always, body on trigger, bundled files only if needed: progressive disclosure.

1 · Where skills live — the cloud-vs-local quirk

Two completely separate stores, and they do not sync. One follows your claude.ai account in the cloud; the other is plain folders on your computer.

The quirk: a skill uploaded to claude.ai does not exist on your machine, and a skill in ~/.claude/skills does not exist in the Chat panel. If a skill "disappears" when you switch panels, this split is why. Keep the canonical copy in a repo; treat any cloud upload as a copy.

2 · How a skill sits in a repo

One folder per skill, lowercase-with-hyphens (gerunds read well: making-handouts, processing-pdfs).

your-project/

.claude/ › skills/

making-handouts/

SKILL.md required — the skill itself
reference.md optional detail, read only when the task needs it
assets/ templates, CSS, fonts the output uses
scripts/ known-good code Claude runs — never loaded as text

inputs/ · operations/ · outputs/ — the rest of the project; the skill travels with the repo

  • SKILL.md is the only required file. Everything else is optional support.
  • Link reference files from SKILL.md ("See reference.md") and keep them one level deep — don't chain file → file → file.
  • scripts/ is for consistency. Code you've gotten working once runs identically every time, instead of being re-invented per session.

3 · Anatomy of the SKILL.md

Two zones: a YAML header between --- lines, then a markdown body.

YAML frontmatter — always loaded

name: making-handouts required · ≤64 chars · lowercase + hyphens
description: Formats class notes into the course handout style. Use when the user asks for a handout, printable, or one-pager. required · ≤1,024 chars · third person · must say what it does and when to use it — this is all Claude sees before deciding to open the skill

Markdown body — loaded only when triggered

  • Keep it under ~500 lines; past that, split into reference files.
  • Assume Claude is already smart. Write only what it doesn't know: your preferences, conventions, domain specifics.
  • Set the degree of freedom. Judgment helps → prose steps. Fragile operation → "run exactly scripts/build.py, do not modify."
  • One term per concept, no time-sensitive instructions.
  • Iterate. Watch what Claude does with it; open it, argue with it, edit.

Rule of thumb: paste the same prompt twice → make it a skill. The skill needs the same code twice → give it a script. The body outgrows a screen or two → split into reference files. The header is the hook; everything else is staged so it costs nothing until used. Vague descriptions ("helps with documents") are the #1 reason skills never fire.

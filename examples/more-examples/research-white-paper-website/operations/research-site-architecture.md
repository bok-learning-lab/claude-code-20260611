# Operation — Research site as a thin file-browser

*The site's whole architecture in one document. About 150 lines of Next.js code (across [`lib-content.ts`](lib-content.ts), [`app-page.tsx`](app-page.tsx), [`browse-page.tsx`](browse-page.tsx), and [`Markdown.tsx`](Markdown.tsx)) turns a research-in-progress folder into a navigable static site. **The research markdown files are the content; the website is just a friendly viewer.***

---

## The architectural commitment

Most research-publication tools rebuild research as web content — paste your paper into a CMS, copy your notes into a wiki, export to Substack. This project's commitment is the opposite: **the markdown files you wrote for yourself ARE the published content**. The viewer reads them in place.

That commitment shapes every design decision in the site:

- **No content directory.** The five sections (`sources/`, `analysis/`, `drafts/`, `proposal/`, `agent-output/`) live at the **repo root**, where a researcher would put them if there were no website. Adding the site doesn't change where the files go.
- **No frontmatter required.** Markdown files don't need YAML headers. The site renders whatever's there.
- **Relative links work.** `[Pasquinelli](../analysis/theoretical_framework.md#pasquinelli)` resolves the same in your editor and in the rendered site. See the link-rewriting move below.
- **No build step for content.** Edit a markdown file, refresh the page. Static generation is opt-in (`generateStaticParams` returns `[]` by default — pages render at request time).

The whole site is **deliberately thin**. It's a *reading apparatus* over a research folder, not an authoring tool, not a CMS, not a "research publication platform."

## The five sections

The `SECTIONS` array in [`lib-content.ts`](lib-content.ts) declares the navigable roots:

```typescript
export const SECTIONS = [
  { slug: "sources",      label: "Sources",      blurb: "Primary texts and PDFs..." },
  { slug: "analysis",     label: "Analysis",     blurb: "Research findings..." },
  { slug: "drafts",       label: "Drafts",       blurb: "Working paper drafts..." },
  { slug: "proposal",     label: "Proposal",     blurb: "Experimental proposal." },
  { slug: "agent-output", label: "Agent output", blurb: "Drafts, syntheses, and bibliographies generated by AI agents." },
];
```

This is the entire navigation. The landing page renders the repo's `README.md` plus a card per section. Each card links to `/browse/<slug>/`, where the dynamic `[...path]` route takes over.

Five sections is the right number for this research. A different research project might use four or seven. The pattern is *small set of named roots*, not *deeply nested taxonomies*.

## Safe-prefix routing — the file-access boundary

The `[...path]` route accepts an arbitrary path (`/browse/analysis/candidate_remedies.md`, `/browse/drafts/paper1_ethnopharmacology.md`, etc.) and reads the file from disk. The hard security constraint: **the request must not be able to read anything outside the declared sections.**

That's enforced by the `ensureSafe()` function in [`lib-content.ts`](lib-content.ts):

```typescript
const SAFE_PREFIXES = SECTIONS.map((s) => s.slug);

function ensureSafe(parts: string[]): string {
  if (parts.length === 0) throw new Error("empty path");
  if (!SAFE_PREFIXES.includes(parts[0])) throw new Error("forbidden");
  const full = path.normalize(path.join(ROOT, ...parts));
  const sectionRoot = path.join(ROOT, parts[0]);
  if (!full.startsWith(sectionRoot)) throw new Error("traversal");
  return full;
}
```

Three checks:

1. **Non-empty path.**
2. **First part is one of the declared section slugs.** A request for `/browse/secrets/api-key.txt` fails on this check — `secrets/` is not in `SAFE_PREFIXES`.
3. **After normalization, the resolved path stays under the section root.** A request for `/browse/sources/../../etc/passwd` fails on this check — even though `sources/` is a declared section, the normalized path leaves the section root.

Every file-read function in `lib-content.ts` (`listDir`, `readText`, `readBinary`, `statPath`) routes through `ensureSafe()`. There is no path that reaches the filesystem without that check.

This is the *one* hard piece of code in the otherwise-thin viewer. Get this wrong and the site exposes the whole filesystem; get this right and the site can serve anything under the declared sections without further security ceremony.

## What the browse route renders, by file type

The `[...path]` route in [`browse-page.tsx`](browse-page.tsx) dispatches based on whether the path is a directory or a file:

- **Directory** → a sorted list (folders first, alphabetical within kind), each entry with its size (for files) or a `▸` marker (for folders). Clicking an entry navigates one level in. Breadcrumb navigation at the top.
- **`.md`** → rendered by the [`Markdown`](Markdown.tsx) component. GitHub-flavored markdown via `react-markdown` + `remark-gfm`. Relative links are rewritten (see below). The page is a single `<div class="prose">`.
- **`.txt`, `.json`, `.ts`, `.tsx`, `.js`, `.py`** → rendered in a `<pre>` block with monospace font and word-wrap. The whole file is shown; large text files load *as text*, no pagination.
- **`.pdf`** → rendered in an `<iframe>` via `/api/file/[...path]` (a thin serverless proxy that returns the raw bytes with the right content type). The PDF fills 85vh.
- **Anything else** → a "No inline preview" message with a download link.

This dispatch table is the second piece of substance after `ensureSafe()`. It's short by design — five file-type branches cover most research artifacts, and "anything else" gets a download fallback.

## Relative-link rewriting in markdown

The single design move that makes editor-fidelity work: the [`Markdown`](Markdown.tsx) component intercepts every `<a>` tag and rewrites relative paths to the site's URL space.

```typescript
function rewriteRelativeLink(href: string, currentParts: string[]): string {
  if (/^[a-z]+:\/\//i.test(href) || href.startsWith("#") || href.startsWith("mailto:")) return href;
  if (href.startsWith("/")) return href;
  const dir = currentParts.slice(0, -1);
  const parts = href.split("/").filter(Boolean);
  const stack = [...dir];
  for (const p of parts) {
    if (p === "..") stack.pop();
    else if (p !== ".") stack.push(p);
  }
  return `/browse/${stack.join("/")}`;
}
```

A markdown file at `analysis/candidate_remedies.md` containing `[theoretical framework](theoretical_framework.md)` produces a link to `/browse/analysis/theoretical_framework.md`. A link `../sources/old_english_herbarium.txt` from the same file produces `/browse/sources/old_english_herbarium.txt`. The links work *the same way* in your editor (resolved against the file's location) and in the browser (resolved by the rewriter).

External links (`http://`, `https://`, `mailto:`) pass through unchanged and open in a new tab. Absolute paths (`/something`) pass through unchanged.

This is what lets the markdown be authored *for the editor first*. Researchers write `[see this](../analysis/candidate_remedies.md)` because that's what makes sense to them while writing; the rewriter makes the same link work on the web.

## What this operation deliberately doesn't do

- **No search.** Five sections of medium-sized markdown isn't a search problem — readers navigate by section and skim files. Add full-text search when the corpus is much larger; until then it's noise.
- **No tags, no categories, no per-page metadata.** The filename and the section are the metadata. Adding a YAML frontmatter system would invite the kind of authoring overhead this design refuses.
- **No comments, no annotation layer.** This is a viewer for *the researcher's own work*. Public-facing annotation is a different feature with different requirements.
- **No analytics.** (Vercel basic deployment metrics may be on.)
- **No editing in the browser.** Edits happen in the editor and land via git. The site is read-only.
- **No build-time static generation.** `generateStaticParams` returns `[]`. Pages render at request time. Adding statics for popular paths is straightforward but unnecessary at this scale.
- **No theme switcher.** The site has a single typographic identity (the `--color-paper` / `--color-rule` / `--color-accent` CSS variables in `globals.css`). The look is deliberately academic-paper-ish: serif headings, narrow column, warm cream background.

## Adapting this to another research project

The pattern ports almost verbatim:

1. **Edit the `SECTIONS` array** in [`lib-content.ts`](lib-content.ts) with your project's section roots. Common adjacent sets: `{ raw-data, analysis, drafts, manuscripts, peer-review }`; `{ archives, transcripts, codings, manuscripts, supplementary }`; `{ interviews, fieldnotes, analysis, drafts, video }`.
2. **Make sure each section is a real folder** at the repo root, with whatever markdown / text / PDF files you have there.
3. **Adjust the landing page's typography** to taste in `globals.css`. The default is academic-cream.
4. **Deploy to Vercel.** No env vars required; the site is fully static-able.

The 150 lines of Next.js code don't change. The substance is the research; the architecture is small and reusable.

## Hard constraints (these survive translation)

- **Safe-prefix routing is non-negotiable.** Every file-read must go through `ensureSafe()`. Don't add a back-door reader function that skips the check.
- **The repo root is the navigation root.** Section folders live at the repo root, not in `_content/` or `content/` or anywhere else. The researcher reads and edits the same paths that get URLs.
- **Relative links resolve the same in the editor and the browser.** This is what makes editor-first authoring work. The Markdown component's link rewriter is the load-bearing piece.
- **No frontmatter required.** Markdown files render as-is. Don't introduce a frontmatter requirement that breaks "open in editor and edit."
- **Read-only by default.** No edit UI. The researcher edits in their editor; the site reads.
